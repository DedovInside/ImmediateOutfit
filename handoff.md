# ImmediateOutfit Handoff

## 1. Что это за проект

`ImmediateOutfit` — учебный продукт по дисциплине `Управление продуктом`.

Формат продукта:
- Telegram-бот для быстрого подбора одежды под ситуацию.

Основная продуктовая идея:
- помочь пользователю быстро решить `что надеть`, не тратя лишнее время утром или перед событием;
- учитывать контекст дня: событие, погоду, активность, стиль, бюджет;
- постепенно перейти от простого MVP к продукту с метриками, циклами улучшений и материалами для защиты.

Целевая аудитория:
- 17-25 лет;
- студенты и молодые специалисты;
- ограниченный бюджет;
- важны уместность, уверенность, скорость и простота интерфейса.


## 2. Что уже реализовано

В репозитории уже внедрен расширенный `bot-first MVP+`:

### Продуктовые сценарии
- `Подобрать образ` — основной flow через анкету.
- `Быстрый подбор` — сокращенный flow для повторных пользователей с профилем.
- `Подобрать под мою вещь` — подбор образа вокруг вещи пользователя.
- `Проверить мой образ` — структурированный текстовый разбор текущего лука.
- `Premium` — заглушка для сбора интереса к будущим платным функциям.

### Данные и архитектура
- SQLite-хранилище вместо in-memory.
- Сущности:
  - `users`
  - `profiles`
  - `saved_outfits`
  - `events`
- Каталог образов вынесен в отдельный слой.
- Добавлен curated-слой с референсами, объяснениями и артикулами.

### Аналитика и курс
- Логируются ключевые события продукта.
- В `main.py` есть dashboard и JSON-метрики.
- Подготовлены документы:
  - `docs/metrics.md`
  - `docs/sprint_map.md`
  - `docs/roadmap.md`
  - `docs/unit_economics.md`

### Тесты
- Есть smoke/unit тесты:
  - `test_recommender.py`
  - `tests/test_recommender.py`
  - `tests/test_storage.py`

## 4. Что важно знать следующему агенту

### Главный контекст
- Проект не просто “про код”, а про закрытие требований курса:
  - MVP
  - метрики
  - 6+ спринтов
  - дорожная карта
  - бизнес-модель
  - отчет и презентация

### Уже принятые продуктовые решения
- Основной интерфейс: `Telegram-бот`.
- `Mini app` отложен.
- `AI try-on` не входит в обязательный ближайший контур.
- Ставка сделана на:
  - быстрый подбор;
  - полезность;
  - доверие к рекомендациям;
  - измеримость через метрики.

### Ограничения и риски
- На машине было очень мало свободного места; это уже мешало установке зависимостей и записи временных файлов.
- Нужно осторожно работать с окружением и не плодить лишние большие директории.
- Референсы и артикулы пока заполнены только частично, как стартовый curated-набор.
- Реальная погодная интеграция пока не внедрена: в продукте есть выбор погоды пользователем, но нет полноценного внешнего weather API flow.

Рекомендуемый вариант — отдельное окружение на `Python 3.12`.

```bash

```

Если `git status` засоряется виртуальным окружением:

```bash

```

### Goal 1. Довести проект до уверенного demo-ready состояния

### Goal 2. Усилить продуктовую полезность

- добавить больше реальных артикулов/ссылок;
- улучшить объяснения рекомендаций;
- при необходимости добавить более явную работу с цветами.

### Goal 3. Подготовить почву для следующих учебных спринтов
- собирать реальные данные по метрикам;
- фиксировать гипотезы и результаты тестов на пользователях;
- готовить материалы для отчета и презентации параллельно с доработкой продукта.


## 8. Что осталось реализовать

### Высокий приоритет
1. Полный ручной e2e QA после установки зависимостей.
2. Проверка реального запуска `bot.py` и `main.py` в чистом `python3.12` окружении.
3. Расширение `data/curation.json`:
   - больше мужских образов;
   - больше женских образов;
   - больше сценариев: учеба, прогулка, свидание, мероприятие, смешанный день.
4. Реальные артикулы и ссылки вместо частично заглушечного наполнения.

### Средний приоритет
5. Интеграция реальной погоды:
   - либо через OpenWeatherMap;
   - либо через другой простой weather API;
   - с fallback на ручной выбор погоды.
6. Улучшение цветовой логики:
   - цвет как отдельный параметр анкеты;
   - теги цветов в образах;
   - более точные советы в сценарии `Проверить мой образ`.
7. Больше сигналов удовлетворенности:
   - анализ `result_feedback`;
   - анализ `review_feedback`;
   - возможно, короткий фидбек “что не понравилось”.

### Низкий / следующий этап
8. Photo-review как экспериментальная функция.
9. Более явный premium flow:
   - waitlist;
   - paywall-заглушка;
   - пробный premium оффер.
10. Mini app и AI try-on — только после подтверждения нужности и если останется ресурс.

## 10. Что не нужно ломать без причины

- Не убирать SQLite-слой.
- Не возвращать in-memory хранение.
- Не переводить проект в mini app без отдельного решения команды.
- Не добавлять тяжелую AI-примерку раньше, чем будет закрыта core value.
- Не превращать продукт в рекламный каталог покупок — интервью скорее подтверждали полезный подбор, а не витрину.


## 11. Полезные ориентиры для следующего агента

- Техническая цель: сделать проект стабильным, демонстрируемым и удобным для полевых тестов.
- Продуктовая цель: доказать, что бот помогает быстрее и увереннее собирать образ.
- Учебная цель: собрать артефакты и метрики так, чтобы команда могла уверенно защитить проект.
